# HCM系统更新计划与排期会议纪要

## 会议概况 (Basic Info)
- **会议主题**：HCM系统4-5月更新排期对齐
- **参会人员**：产品/业务负责人、技术团队负责人、修正（后端开发）等（根据上下文推断）
- **关键摘要**：本次会议旨在梳理并确定HCM系统近期各项高优更新需求的开发排期。会议明确了4月至5月的核心交付任务，其中修复补差额HC科目错误为所有数据导出的前提，审批流（含平行审批及邮件通知）与报表导出功能（现状及申请后双按钮）被列为最高优先级。同时，会议决定将部门级预算审批流与移动端界面大改等低频需求延后至6月处理。

## 核心议题与讨论 (Key Discussion Points)

### 1. 数据准确性与补差额科目修复（最高优前提）
- **现状/背景**：历史补明细数据存在偏差，导致财务与HR数据对账困难。目前补差额HC的所有非工资科目（如公积金等）被错误地归入了“保险”科目。
- **主要观点**：该问题若不修复，后续导出的预算明细与变更日志数据均将报错，必须作为首要任务解决。
- **结论**：将“修复补差额HC科目映射规则”定为最高优先级，优先于其他所有导出与日志功能开发。

### 2. 审批流调整与单据状态通知
- **现状/背景**：
  1. 当前容与sam的平行审批缺乏通知机制（一方驳回，另一方不知情）。
  2. 审批单据在移动端（OTP）被拒绝或完成后直接消失，用户无法追踪状态。
- **主要观点**：
  - 业务侧希望在OTP内保留“已完成”或“已拒绝”状态，但技术侧评估当前前端框架实现成本过高。
  - 妥协方案：短期内通过发送邮件或在PC端增加状态展示来解决通知问题。
- **结论**：四月份必须上线“审批流节点调整（涉及刘伟、袁俊）”及“平行审批驳回邮件通知功能”。OTP单据状态前端优化留待7月后视情况规划。

### 3. 预算变更日志与“项目标签”修改管控
- **现状/背景**：目前部门级预算可以直接在前台修改生效，缺乏审批流与变更日志；“项目标签”也允许直接编辑。
- **主要观点**：
  - 业务侧（文佳怡等）关注项目标签的修改审计，认为不应脱离审批独立存在。
  - 技术侧认为开发完整的部门级预算审批流耗时较长（涉及移动端字段拆分），建议延后。
- **结论**：暂时关闭前端直接修改“项目标签”的入口；部门级预算暂不强制加审批，短期内通过后台增加简单的修改Log（记录谁、何时、改了什么）作为过渡。

### 4. 报表导出功能设计确认
- **现状/背景**：需提供支持二次加工（如数据透视）的部门及HC预算导出报表。
- **主要观点**：
  - **科目规则**：对于部分部门不存在的科目（如工会经费、绩效浮动），统一规范为显示“0”。
  - **导出时机**：为满足不同场景，需要提供“导出现状”与“导出申请后”两个数据版本的下载按钮。
  - **项目标签展示**：项目标签作为文本字段，按列横向铺开（1-12月展示对应的项目编码），以支持按项目的汇总透视。
- **结论**：按照统一的扁平化表样进行开发，报表生成按钮拆分为“现状”与“申请后”，目标在2026-05-20（520）前提供可测试版本。

### 5. 预算看板与移动端界面优化
- **现状/背景**：用户需要预算与实际消耗的对比看板；移动端审批界面未拆解“其他人员”的细项。
- **主要观点**：看板需求（加预算、实际、差异三列）逻辑简单；但移动端审批界面的数据结构改动巨大，且当前工作日严重不足（4月仅剩11个工作日）。
- **结论**：预算看板视开发资源穿插进行；移动端审批界面重构随部门级预算审批流一并延后至6月或以后。

## 决议与共识 (Decisions)
1. **优先保障数据底座**：立即修复补差额HC科目错误问题，确保后续导出的所有明细与日志数据准确。
2. **四月核心交付**：锁定完成“审批节点调整”、“平行审批及邮件通知补充”，并关闭项目标签的直接编辑入口。
3. **报表交付节点**：确认导出报表（含现状与申请后双版本、工会经费等补零规则）于2026-05-20前提交测试。
4. **非紧急需求延后**：明确放弃在4-5月进行移动端审批界面大改及部门级预算审批流开发。

## 风险与遗留问题 (Risks & Parking Lot)
1. **财务科目拆分风险**：预计7月份财务可能会将“商业保险”拆分为“补充医疗”和“福利收入”（需纳税），届时HCM系统需配合进行底层科目结构改造，方案需在6月底前与财务完全确认。
2. **移动端单据状态显示盲区**：目前采用邮件通知作为折中方案，但长远看，OTP/移动端内单据在被拒绝后直接消失仍存在体验缺陷，需在7月后重新规划前端状态展示方案。
3. **项目过渡期HC挂起**：关于“项目过渡期人员预算挂靠”的需求因逻辑过于复杂且不确定因素多，本次会议决定先行挂起，待真实业务场景发生时再做讨论。
