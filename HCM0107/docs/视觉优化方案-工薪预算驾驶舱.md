# 工薪预算驾驶舱 - 「当前工薪执行现状及现行规划」视觉优化方案

## 问题分析

### 当前状态
「当前工薪执行现状及现行规划」表格存在严重的信息过载问题：

1. **列数过多**：表格包含30+列，横向滚动距离长
2. **视觉层级不清晰**：2026年和2027年数据结构重复，难以快速区分
3. **数据密度过高**：HC数据的5个员工类型（正编、实习、劳务派遣、人力外包、兼职）在两个时间维度下重复展示
4. **认知负荷高**：用户需要在大量相似数据中快速定位关键信息

### 现有优化尝试
- 已使用不同背景色（灰色#f7f9fc vs 蓝色#e6f2ff）区分两年
- 已添加"一键折叠/展开"功能
- 已支持方案切换（UI中已预留方案1-3切换按钮）

## 方案一：视觉层级增强方案（列分组强化）

### 设计原理

基于**格式塔心理学中的接近性和相似性原则**，通过强化视觉层级和列分组，降低用户的认知搜索成本。

### 核心策略

#### 1.1 三级分组视觉系统

```
第一级：年份分组（2026年 | 2027年）
        └─ 背景：明显的色彩区块 + 粗分隔线

第二级：功能模块分组（HC | 工薪成本）
        └─ 分隔：中等粗细分隔线 + 微妙背景差异

第三级：时间维度分组（当前/年末目标 | 年初计划/年末目标）
        └─ 分隔：细线 + 字体粗细对比
```

#### 1.2 视觉分隔增强

| 分隔层级 | 视觉处理 | CSS实现 |
|---------|---------|---------|
| 年份间 | 3px实线 + 8px留白 | `border-right: 3px solid #0052D9; margin-right: 8px;` |
| 功能模块间 | 2px实线 | `border-right: 2px solid #C9CDD4;` |
| 时间维度间 | 1px虚线 | `border-right: 1px dashed #E5E6EB;` |
| 员工类型间 | 0.5px浅灰线 | `border-right: 1px solid #F2F3F5;` |

#### 1.3 列头设计优化

- **年份列头**：使用品牌色背景 + 白色粗体文字
- **功能模块列头**：使用深灰色背景 + 白色文字
- **员工类型列头**：添加图标标识（正编👤、实习🎓、派遣🔄、外包📦、兼职⏱️）

#### 1.4 数据单元格视觉提示

- **关键数据**（全年预算、年末目标）：使用粗体
- **次要数据**（当前HC、年初计划）：使用常规字体
- **变化数据**（预实比对中的变化）：添加颜色编码（红涨绿跌）

## 方案二：渐进式信息披露方案（默认折叠 + 按需展开）

### 设计原理

基于**渐进式披露（Progressive Disclosure）**原则和**认知负荷理论**，默认只展示关键指标，允许用户按需展开详细信息。

### 核心策略

#### 2.1 三级信息架构

```
Level 1: 关键指标（默认展开）
├─ 部门名称
├─ 2026年：HC总数、工薪成本总额、预实比对状态
└─ 2027年：HC总数、工薪成本总额

Level 2: 核心数据（点击展开）
├─ HC详情（当前 vs 目标）
├─ 工薪成本明细（累计、预算、剩余）
└─ 变化趋势（Δ值、百分比）

Level 3: 完整数据（再次展开）
├─ 5种员工类型完整数据
├─ 12个月月度趋势
└─ 历史对比数据
```

#### 2.2 默认视图设计

| 列 | 内容 | 宽度 |
|----|------|------|
| 部门 | 部门名称 | 200px |
| 26年HC | 📊 152人 | 100px |
| 26年成本 | 💰 2,400万 | 120px |
| 26年状态 | ⚠️ 预警 | 80px |
| 27年HC | 📊 165人 | 100px |
| 27年成本 | 💰 2,520万 | 120px |
| 操作 | [展开] ▼ | 80px |

#### 2.3 展开视图设计

**二级展开**（点击"展开"按钮）：
- 显示HC总数对比（当前 vs 目标）
- 显示成本三件套（累计、预算、剩余）
- 添加迷你趋势图（sparkline）

**三级展开**（点击HC或成本标题）：
- 展开5种员工类型数据
- 使用手风琴效果，一次只展开一个模块

#### 2.4 视觉提示系统

- **可展开标识**：列标题后添加 🔽 图标
- **已展开标识**：列标题后添加 🔼 图标
- **数据状态指示**：
  - 🟢 正常（偏差<5%）
  - 🟡 预警（偏差5-10%）
  - 🔴 风险（偏差>10%）

```javascript
// 计算属性：生成摘要数据
const summaryData = computed(() => {
  return store.treeData.map(dept => {
    const hc26Cur = dept.values.h26_cur.reduce((a, b) => a + b, 0)
    const hc26Tgt = dept.values.h26_tgt.reduce((a, b) => a + b, 0)
    const hc27Plan = dept.values.h27_plan.reduce((a, b) => a + b, 0)

    const cost26Used = dept.values.cost_acc
    const cost26Total = dept.values.cost_year
    const cost27Total = dept.values.cost_year * 1.05

    const status26 = cost26Used / cost26Total
    let status = 'normal'
    if (status26 > 1.1) status = 'risk'
    else if (status26 > 1.05) status = 'warning'

    return {
      ...dept,
      summary: {
        hc26Cur,
        hc26Tgt,
        hc27Plan,
        cost26Used,
        cost26Total,
        cost27Total,
        status
      }
    }
  })
})

// 展开状态管理
const expandedRows = ref<Set<string>>(new Set())
const expandedModules = ref<Record<string, string>>({})

const toggleRow = (rowId: string) => {
  if (expandedRows.value.has(rowId)) {
    expandedRows.value.delete(rowId)
  } else {
    expandedRows.value.add(rowId)
  }
}

const toggleModule = (rowId: string, module: string) => {
  const key = `${rowId}-${module}`
  if (expandedModules.value[key] === module) {
    delete expandedModules.value[key]
  } else {
    expandedModules.value[key] = module
  }
}
```

## 方案三：卡片式响应布局方案

### 设计原理

基于**响应式设计**和**移动优先**理念，将表格数据转换为卡片式布局，适应不同屏幕尺寸，提升移动端体验。

### 核心策略

#### 3.1 卡片层级结构

```
部门卡片（Level 1）
├─ 卡片头部：部门名称 + 状态标签
├─ 快速概览：关键指标仪表盘
├─ 2026年数据区（可折叠）
│   ├─ HC卡片组（5个员工类型小卡片）
│   └─ 成本卡片组（累计、预算、剩余）
└─ 2027年数据区（可折叠）
    ├─ HC卡片组
    └─ 成本卡片组
```

#### 3.2 卡片设计规范

**部门卡片**：
- 尺寸：根据屏幕自适应（桌面: 1200px | 平板: 768px | 手机: 100%）
- 阴影：`box-shadow: 0 2px 8px rgba(0, 0, 0, 0.08)`
- 圆角：`border-radius: 8px`
- 内边距：`padding: 20px`

**数据小卡片**（员工类型）：
- 尺寸：140px × 100px
- 布局：Grid 2列或3列（响应式）
- 内容：
  ```

  │  正编  👤   │
  │  ─────────  │
  │  当前: 50   │
  ```
