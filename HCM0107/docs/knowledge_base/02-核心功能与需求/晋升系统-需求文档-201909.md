1. 需求概述
晋升系统旨在帮助提升晋升管理效率，提升员工体验。本期研发的主要功能包括晋升申报，晋升申报审
批，晋升结果审批；管理自己的晋升，下属的晋升等功能。
2. 晋升系统流程
2.1 流程说明
2.1.1 流程图

2.1.2 申报期
 晋升开始前，OD在后台创建一次晋升，创建后点击“开启晋升”，晋升开启后即进入晋升申报
阶段。
 晋升申报期间，首先由HRG提交申报名单，提交后到HRD确认，HRD确认（提交）前HRG

还可以继续补提交部门内的晋升申报人员。如果有 HRA，则在 HRA 确认之前可以补充申报。
 HRD或HRA提交到二级部门/直属三级/特殊独立的三级部门审批。部门负责人审批时可退回。
 各部门审批后汇总到OD审批，OD需按业务进行审批，OD审批后提交至HRSVP和业务负责
人（如鲁总）审批。
 HRSVP和业务负责人审批时可以退回，退回后退至上一个审批人。
 业务负责人审批结束后，OD操作开启晋升评审。
2.1.3 评审期
 OD开启晋升评审后，参与评审的员工收到通知，可以开始准备评审。
 对于晋升 14 级以下的员工，需要填写晋升考察表，填写后由直接上级确认。然后 HRD 维护
考察结果，并结束整个部门的考察。HRD结束考察前，HRG可以维护考察结果，也可以不维
护。
 对于晋升14级以上的员工，需要在系统中填写晋升申请档案，并在系统中下载面审模板，准
备参加面审。面审结束后由OD在后台导入面审结果。
2.1.4 审核期
 晋升考察和面审全部结束后，OD 统一按业务发起审批，需 HRSVP 和业务负责人审批。审批
内容包括通过晋升评审和未通过晋升评审的人员。
 审批结束后七天，系统触发晋升结果生效。
2.1.5 通告期
 晋升结果生效后，系统自动触发相关通知给员工、三级以上部门负责人以及HR。

2.3 人员及组织架构更新规则
2.3.1 人员更新规则
➢ 晋升开启后，该次晋升的人员基数不再发生变化。即所有入职的人员不再进入该次晋升；如
发生离职，仍作为该次晋升的基数，但需要标记出。
➢ 晋升结束后，数据落地时，如在晋升生效日期前的离职，则该数据不落地；如在晋升生效日
期后离职，则该数据落地。
2.3.2 组织架构更新规则
➢ 晋升开始后，部门发生变化，人员发生调动，不更新数据；
➢ 晋升开始后，如部门负责人发生变化，则需要更新部门负责人。
