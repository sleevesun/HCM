组织资源划转功能设计规范书 (V1.2)
1. 业务背景与产品定位
• 痛点：在“业务剥离”或“组织拆分”场景中，业务部门需要在老部门正式撤销前，先行将大部分预算和空编制（HC）转移至新部门开展工作。现有的“撤销自动触发”逻辑无法满足这种时间差需求。
• 定位：提供一个主动的、灵活的资源调拨工具。
• 核心隐喻：银行转账。A账户（老部门）向 B账户（新部门）手动划转资金和资产，转账后 A账户依然存活，只是余额减少。
2. 核心业务流程
1. 发起：HRBP 或 组织发展专家 (OD) 发起《资源划转申请》。
2. 配置：选择转出部门、接收部门、生效日期。
3. 选择：
    ◦ 勾选需要转移的 空缺 HC（按人头移交）。
    ◦ 输入需要划转的 部门级预算金额（按金额移交）。
    ◦ [新增] 输入需要划转的 补差额 HC 预算（按员工类型金额移交）。
4. 校验：系统执行余额校验及接收方属性校验。
5. 审批：经过 财务BP (FBP) 及 双方部门负责人审批。
6. 生效：数据实时划转，生成财务凭证或调整记录。
3. 界面交互设计 (UI/UX)
页面标题：新建资源划转单
布局风格：通栏卡片式布局，强调“从左至右”的流转感。
3.1 整体布局结构
页面分为三个垂直区域：
1. 基础信息配置区 (Header)：确定“谁转给谁”以及“什么时候生效”。此区域锁定后，下方数据才会加载。
2. 资源操作区 (Body)：核心作业区，采用 Tab 页签切换“资金”、“编制”与“补差额”。
3. 底部操作栏 (Footer)：包含“保存草稿”、“提交审批”及实时汇总信息。
3.2 基础信息配置区 (顶部)
此区域背景为浅灰色，作为全局过滤器使用。
• [第一行] 划转对象配置
    ◦ 转出部门 (Source)：[ 下拉搜索框 ]
        ▪ 交互：选择后，自动加载该部门当前的预算余额和空 HC 列表。
        ▪ 校验：仅展示状态为“生效中”的部门。
    ◦ 接收部门 (Target)：[ 下拉搜索框 ]
        ▪ 交互：选择后，系统即时校验该部门属性（实体/非实体）。若为非实体部门，显示黄色警示条：“接收方为非实体节点，仅可划转资金，不可划转 HC 及 补差额预算”。
        ▪ 互斥：不可与转出部门相同。
    ◦ 生效日期：[ 日期选择器 ] (默认显示今日，允许选择未来日期)
• [第二行] 业务信息
    ◦ 划转原因：[ 文本域 ] (必填，限200字。Placeholder：“请填写组织变革背景或相关会议纪要编号”)
3.3 核心操作区 (Tab页签)
Tab 1：部门级预算 | Tab 2：空缺 HC | Tab 3：补差额 HC 预算
【Tab 1：部门级预算交互详情】
布局分为三栏：资产看板 (左) -> 划转动作 (中) -> 模拟预览 (右)
• 左侧：资产看板 (Source View)
    ◦ 展示转出部门的当前账户状态。
    ◦ 样式：卡片视图，包含三个关键指标。
        ▪ 总预算：¥ 2,000,000 (灰色数字)
        ▪ 已使用：¥ 800,000 (灰色数字)
        ▪ 当前可用余额：¥ 1,200,000 (高亮大字，绿色)
    ◦ 异常态：若余额 < 0，显示红色数字，并加注标签“已超支”。
• 中间：划转动作区 (Action Zone)
    ◦ 划转金额输入框：[ ¥ 输入金额 ]
        ▪ 校验规则：
            1. 只能输入正数，最多两位小数。
            2. 实时校验：输入值 > 左侧可用余额时，输入框变红，下方提示“超出当前可用额度”。
            3. 负数拦截：若左侧余额已超支，此输入框默认禁用 (Disabled)，并显示 Tooltip “账户超支，无法划转”。
    ◦ 快捷按钮：[ 全部划转 ] (点击后自动填入当前可用余额的数值)
    ◦ 流向箭头：视觉元素，一个巨大的向右箭头 (➔)，增强流转感。
• 右侧：模拟预览区 (Simulation View)
    ◦ 展示接收部门在划转后的预期状态。
    ◦ 列表项：
        ▪ 接收前余额：¥ 50,000
        ▪ + 本次转入：¥ 800,000 (蓝色高亮)
        ▪ = 接收后余额：¥ 850,000
    ◦ 意图：帮助用户二次确认，防止看错单位（如把 10万 输成 100万）。
【Tab 2：空缺 HC 交互详情】
布局采用经典的 穿梭框 (Shuttle Box) 设计。
• 全局阻断逻辑：
    ◦ 若在顶部选择了“非实体部门”作为接收方，此 Tab 页签直接置灰不可点，并显示蒙层提示：“接收方属性不支持承接编制”。
• 左侧列表：待划转 HC 池 (可多选)
    ◦ 顶部筛选：[ 职位族下拉 ] [ 职级下拉 ] [ 搜索框 ]
    ◦ 表头：| 勾选 | HC编码 | 职位名称 | 职级 | 预估年成本 |
    ◦ 数据源：仅加载转出部门下状态为“空缺 (Vacant)”的 HC。
    ◦ 底部统计：共 15 个空缺，已选 0 个。
• 中间：操作按钮组
    ◦ [ > ]：将左侧勾选项移入右侧 (启用条件：左侧至少选中一项)。
    ◦ [ >> ]：全部移入右侧。
    ◦ [ < ]：将右侧勾选项移回左侧。
    ◦ [ << ]：全部移回左侧。
• 右侧列表：拟划转清单
    ◦ 表头：| 移除 | HC编码 | 职位名称 | 职级 | 预估年成本 |
    ◦ 底部统计：本次拟划转：3 人，涉及预算包：¥ 1,500,000 (动态求和)。
【Tab 3：补差额 HC 预算交互详情】(新增)
布局采用 列表映射 (Mapping List) 设计，强调“类型对齐”与“金额合并”。
• 全局阻断逻辑：同 Tab 2，若接收方为“非实体部门”，此 Tab 置灰不可用。
• 表头设计：
| 员工类型 (Key) | 转出方余额 (Source) | 本次划转金额 (Action) | 接收方状态预览 (Target Preview) |
• 行内容交互：
    ◦ 行 1：正编补差
        ▪ Source：¥ 500,000
        ▪ Action：[ 输入框 ] (例如输入 300,000)
        ▪ Target：
            • 若接收方已有同类型补差额 HC -> 显示图标 🔄 [合并金额] (余额: ¥10w + 30w = 40w)
            • 若接收方无同类型补差额 HC -> 显示图标 ✨ [自动创建] (余额: 0 + 30w = 30w)
    ◦ 行 2：实习生补差
        ▪ Source：¥ 20,000
        ▪ Action：[ 输入框 ]
        ▪ Target：(实时计算预览)
• 底部提示：
    ◦ ℹ️ 提示：若接收部门不存在对应类型的补差额 HC，系统将自动创建一个新的补差额虚拟 HC 并转入金额。
3.4 底部操作栏 (Footer)
• 左侧信息：
    ◦ 摘要：部门级预算 ¥ [金额] + HC [数量] 个 + 补差额预算 ¥ [金额]
• 右侧按钮：
    ◦ [ 取消 ]：返回上一页。
    ◦ [ 保存草稿 ]：保存当前状态，不触发校验。
    ◦ [ 提交审批 ]：
        i. 触发全页面数据校验。
        ii. 再次弹窗确认：“您确定要将 [转出部门] 的资源划转给 [接收部门] 吗？操作生效后不可撤销。”
        iii. 提交成功，跳转至详情页。
4. 功能逻辑详情
4.1 基础校验规则
1. 互斥校验：转出部门与接收部门不能相同。
2. 状态校验：双方部门必须均为 Active (生效中) 状态。冻结或拟撤销部门不可作为接收方。
4.2 部门级预算处理逻辑
• 上限控制：划转金额 ≤ 转出方当前可用余额 (Rem)。
    ◦ 解释：不允许透支划转。如果老部门想把超支的烂账甩给新部门，系统应拦截，提示“余额不足”。
• 负数拦截：若转出方当前余额为负数（已超支），则锁定输入框，禁止划转金额。提示：“该部门已超支，无可转移预算”。
4.3 HC 划转处理逻辑
• 非实体部门阻断 (核心风控)：
    ◦ IF 接收方部门属性 = 非实体部门 (x.1级/02级等管控节点)
    ◦ THEN 系统报错拦截：“接收方为非实体部门，无法承接 HC 编制，请重新选择。”
    ◦ 设计意图：与撤销逻辑保持一致，防止 HC 流入虚框部门造成数据污染。此为硬性阻断。
• HC 属性清洗：
    ◦ HC 转移后，其 归属部门ID 变更为接收方。
    ◦ HC 的 生效日期、职级、薪资范围 等属性保持不变（原样平移）。
4.4 补差额 HC 划转处理逻辑 (新增)
• 非实体部门阻断：同标准 HC，非实体部门禁止接收补差额预算。
• 自动创建与合并规则：
    ◦ 系统遍历用户输入的每一行（员工类型）。
    ◦ Check：接收部门是否存在该“员工类型”的补差额 HC？
        ▪ YES：执行 Update 操作。接收方余额 = 原余额 + 划转金额。
        ▪ NO：执行 Create 操作。在接收方名下新建一个同类型的补差额 HC，并将划转金额作为其初始余额。
• 余额校验：单行划转金额 ≤ 转出方该类型补差额 HC 的当前余额。
5. 对“部门撤销”逻辑的影响
本功能上线后，与 V5.0 撤销逻辑形成完美闭环：
1. 阶段一（业务拆分期）：
    ◦ 用户使用本功能（资源划转），将老部门 90% 的预算、HC 和补差额预算搬运到新部门。
    ◦ 老部门余额变少，但依然存活，用于处理历史报销。
2. 阶段二（正式撤销期）：
    ◦ 用户发起《部门撤销》。
    ◦ 系统触发 V5.0 撤销逻辑。
    ◦ 此时，系统发现老部门剩余预算极少（如仅剩 ¥5000），于是将这 ¥5000 兜底上移给上级部门。
    ◦ 系统发现老部门无空 HC（因为都转走了），于是无 HC 需要作废。
    ◦ 结果：平稳过渡，财务账目清晰。
