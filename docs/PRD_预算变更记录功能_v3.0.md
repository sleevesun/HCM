# HCM 系统：预算变更记录功能全量详细设计方案 (v3.0)

## 1. 方案背景与目标
### 1.1 业务背景
目前 HCM 系统的预算变更分布在审批、调动、组织调整等多个模块。为了实现“人财对账”的精准性，财务系统需要获取每一笔变更的明细数据（科目级 + 项目标签级）。

### 1.2 核心目标
- **可视化追溯**：在 HCM 系统内为 HR 提供直观的变更流水查询。
- **自动化集成**：将变更日志通过数据中台准实时同步至财务系统。
- **精细化管控**：确保所有预算变动均可下钻至 14 类科目及对应的项目标签。

---

## 2. 模块一：HCM 系统前端展示模块设计

### 2.1 界面原型与样式规范
- **组件选型**：使用 Ant Design Vue 的 Table 组件实现“嵌套表格”结构。
- **交互逻辑**：
    - **默认视图**：展示汇总列表，每行代表一次业务变更。
    - **展开交互**：点击行首的“+”号，在当前行下方展开“变更明细”子表。
    - **视觉反馈**：
        - 数值增加显示为**红色**并带“+”前缀。
        - 数值减少显示为**绿色**并带“-”前缀。
        - 变更类型使用不同颜色的**标签（Tag）**区分（如：调动-橙色、审批-蓝色）。

### 2.2 汇总列表字段定义
| 字段名称 | 数据类型 | 说明 |
| :--- | :--- | :--- |
| 日志编号 | 字符串 | 全局唯一，例：LOG-20260413-001 |
| 发生时间 | 日期时间 | 业务生效的精确时间（秒级） |
| 变更类型 | 标签 | 预算审批、带 HC 调动-调出、带 HC 调动-调入、部门撤销、部门剪切 |
| 涉及部门 | 字符串 | 发生变更的目标部门全称 |
| 类别 | 字符串 | 人员类别（如正编、实习生）或费用类别（如加班费） |
| HC 变动 | 整数/字符 | 编制数量增减，纯费用变更显示为“-” |
| 总额变动 | 货币 | 该笔业务涉及的预算总金额变动（元） |

### 2.3 嵌套明细子表字段定义
当用户展开某条记录时，展示该笔业务拆分后的财务科目分布：
| 字段名称 | 说明 | 示例 |
| :--- | :--- | :--- |
| 财务科目 ID | 财务系统中的标准科目编码 | ACC-001 |
| 财务科目名称 | 14 类科目之一（详见第 4 章） | 工资预算 |
| 项目标签 | 该科目金额归属的具体项目 | 完美世界项目 A |
| 变动金额 | 该科目在该项目下的具体变动值（元） | +150,000.00 |

## 3. 模块二：后台日志同步传输系统设计

### 3.1 传输机制与格式
- **传输媒介**：通过数据中台（Data Mid-end）进行异步中转。
- **数据格式**：标准 JSON 格式。
- **同步频率**：HCM 实时落地日志表，中台每 2 小时抽取增量快照。

### 3.2 传输数据项定义
传输给财务系统的记录包含以下字段：

| 字段名称 | 说明 | 示例 |
| :--- | :--- | :--- |
| 日志编号 | 全局唯一 ID | LOG-20260413-001 |
| 发生时间 | 业务生效时间 | 2026-04-13 10:00:25 |
| 业务类型 | TRANSFER_OUT, APPROVAL 等业务场景枚举 | TRANSFER_OUT |
| 部门代码 | 涉及部门的编码 | DEPT_001 |
| 部门名称 | 涉及部门的名称 | 研发一部 |
| HC 变动 | 编制数量增减（纯费用为 0） | -5 |
| 总变动金额 | 该笔业务的总预算变动（元） | -250000.00 |
| 补差额标识 | **true 表示该记录属于系统补差额，不产生日志且不传输** | false |
| 明细项列表 | 包含具体的科目及项目拆分明细 | [见下表] |

#### 明细项结构：
| 字段名称 | 说明 | 示例 |
| :--- | :--- | :--- |
| 财务科目 ID | 财务系统标准科目编码 | ACC-001 |
| 财务科目名称 | 14 类标准科目名称 | 工资预算 |
| 项目标签 | 变动归属的项目名称 | 项目 A |
| 明细变动金额 | 数组内总和必须等于总变动金额 | -150000.00 |

## 4. 预算科目全量清单
明细数据必须覆盖以下 14 个标准科目：
1. 工资预算
2. 绩效工资
3. 商业保险（补充医疗）
4. 工会经费
5. 饭补
6. 工伤保险
7. 医疗保险
8. 失业保险
9. 养老保险
10. 公积金
11. 签字费
12. 离职补偿金
13. 加班费
14. 其他人力成本预留

---

## 5. 补差额场景的闭环逻辑
- **不记录原则**：当系统因人员入职、调薪自动重算 HC 预算并产生补差额时，后台静默处理。
- **项目标签差异**：系统允许“补差额 HC”的项目标签与主 HC 标签不同，但在计算部门可用余额时会自动对冲，此过程不向外部系统（财务）暴露日志。

---

## 6. 验收标准
1. **前端**：点击“预算变更记录”可看到带颜色区分的增减数值，展开后明细清晰。
2. **后台**：中台资产表中 `items` 数组内的科目明细完整，且 `is_supplement_diff` 过滤逻辑生效。
3. **一致性**：财务系统接收到的总额与 HCM 审批单据总额完全一致。
