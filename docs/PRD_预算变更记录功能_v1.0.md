## 2. 项目背景与目标
### 2.1 背景描述
目前 HCM 系统的预算变更逻辑散落在各个业务模块（如：预算审批、人员调动、组织架构调整）。由于缺乏统一的变更日志，财务系统（TA）无法准确追溯每一笔人力成本变动的具体原因及对应的项目归属。这导致“人财对账”困难，项目成本核算不精准。

### 2.2 核心目标
1.  **全面追溯**：记录每一笔影响 HC 和预算的业务操作，确保审计透明。
2.  **财务对账**：提供明细到“财务科目”的变动数据，支撑财务 TA 系统入账。
3.  **多维管控**：确保所有预算变动均带有“项目标签”，支持按项目维度的成本核算。
4.  **自动化同步**：通过数据中台（Data Mid-end）实现 HCM 日志向财务系统的自动流转。

## 3. 用户场景分析
### 场景 1：年度/期中预算调整
- **角色**：HRD / 预算管理员
- **操作**：在 HCM 发起年度预算增补申请。
- **结果**：审批通过后，系统自动生成一条“预算审批”日志，包含各科目（工资、奖金等）的增量。

### 场景 2：带 HC 跨部门调动
- **角色**：部门经理 / HRBP
- **操作**：将一名员工从 A 部门调动至 B 部门（带 HC 调动）。
- **结果**：系统生成两条对冲日志（A 部门调出，B 部门调入），自动根据该员工的当前薪资科目和项目标签拆分变动额。

### 场景 3：组织架构重组 (部门撤销/剪切)
- **角色**：组织架构管理员
- **操作**：撤销一个尚有空余 HC 的部门。
- **结果**：系统生成“部门撤销”日志，记录该部门失效的 HC 数量及对应释放的预算金额，解决财务侧“HC 数量对不上”的痛点。

## 4. 功能需求详述
### 4.1 日志触发与记录机制
系统必须在以下业务节点自动触发日志记录：
- **预算审批**：审批状态更新为“已通过”时。
- **人员调动**：带 HC 调动生效当刻。
- **部门调整**：部门撤销或在组织树上发生位置转移（剪切）时。

### 4.2 数据结构定义 (Log Data Model)
单条日志必须包含以下三层数据：

#### A. 基础属性
- **日志编号 (Log ID)**: 全局唯一标识 (例: LOG-20260413-001)。
- **发生时间 (Timestamp)**: 精确到秒。
- **变更类型 (Type)**: 预算审批、带HC调动-调出、带HC调动-调入、部门撤销、部门剪切等。
- **涉及部门 (Department)**: 变更发生的部门名称/编码。
- **人员/费用类别 (Category)**: 正编、实习生、劳务派遣、离职补偿金、加班费等。

#### B. 核心指标
- **HC 变动数量**: 整数 (增加为正，减少为负)。对于纯费用变更显示为“-”。
- **总预算变动金额**: 元 (保留两位小数)。

#### C. 财务明细 (Finance Details) - 核心对接项
| 字段 | 说明 | 示例 |
| :--- | :--- | :--- |
| 财务科目 ID | 对应财务系统 TA 的科目编码 | S001 |
| 财务科目名称 | 具体的科目项 | 基本工资 |
| 项目标签 | 归属的项目标签 (必须) | 完美世界项目A |
| 变动金额 | 该科目在该项目下的具体变动值 | +10,000 |

## 5. 交互与 UI 设计
### 5.1 入口设计
- **位置**：工薪预算驾驶舱 (SalaryBudgetCockpit) -> 预实比对弹窗 -> 右上角【预算变更记录】按钮。

### 5.2 列表展示 (嵌套表格)
- **主列表**：展示所有变更记录的汇总信息。
- **详情页 (展开项)**：点击某条日志，展开展示该日志对应的财务明细列表（科目 + 项目标签 + 金额）。

### 5.3 筛选与导出
- **筛选条件**：支持按时间段、部门、变更类型、日志编号搜索。
- **导出**：支持导出为 Excel 报表供财务线下审计。

## 6. 技术集成方案 (数据中台对接)
### 6.1 数据存储
- HCM 建立专用日志表 `hcm_budget_change_log`。

### 6.2 数据分发
- **落地路径**：HCM 实时将日志快照推送至数据中台指定资产：`dw.hcm_budget_change_log_snapshot`。
- **同步频率**：数据中台定时采集（建议 T+0 准实时或每 2 小时一次）。
- **TA 系统接入**：财务 TA 系统订阅该资产，根据自身逻辑实现预算额度的实时核减。

---

## 7. 非功能性需求
1.  **事务一致性**：业务操作成功与日志记录必须在同一个数据库事务中，严禁业务成功但日志丢失。
2.  **性能要求**：单次日志加载时间（分页 20 条）响应时间应小于 500ms。
3.  **安全性**：仅具备“预算管理员”或“财务接口人”权限的用户可见。
4.  **历史留存**：日志数据永久保留，至少保留 3 年在线查询。

2.1 接口一：发起前校验接口（OA发起流程前调用）
1. 接口职责： 在OA端用户点击“提交”时，同步调用此接口，校验部门有效性及当前HC数量是否满足扣减条件。

2. 核心校验逻辑：

- 部门有效性校验 ：校验OA传入的部门编码在HCM中是否存在且处于有效（未撤销）状态。
- HC总量校验（仅限减少HC场景） ：
  - 校验层级 ：HCM需根据OA传入的具体部门（如：星云工作室\策划部），向上追溯至该部门所在的**“预算汇总部门”**（如：星云工作室），并在该汇总部门层级进行总HC数量的校验。
  - 校验规则 ：只校验预算汇总部门的 总HC数量 是否够减（即：当前总HC数 - 申请减少HC数 ≥ 0）。
  - 非强校验说明 ：不校验当前HC是否已被具体外包人员占用（不校验在岗状态），只要总数够减即可放行。
3. 并发与冻结机制：

- 暂不锁定库存 ：为了保证系统性能与流程灵活性，OA审批期间，HCM系统内 不 对涉及的HC或预算做任何冻结/锁定处理。最终的数据一致性由落地接口的强校验来保障。 2.2 接口二：审批后落地接口（OA流程审批归档后调用）
1. 接口职责： OA流程审批通过后，异步调用此接口，将审批结果真正在HCM系统中执行数据变更（增减HC与预算）。

2. 核心处理逻辑（按场景）：

- 场景A：同时增加/减少HC与预算
  
  - 增加HC与预算 ：
    - HC创建 ：按照“项目×HC数量”的维度，在HCM中新增相应数量的HC编制。
    - 预算分配 ：将新增的总预算在各个新增的HC上进行 平分 。
    - 余数处理规则 ：若出现预算除不尽的情况（例如增加3个HC，总预算增加10000元），保留两位小数，将除不尽的 余数全部挂在最后一个HC上 （即：HC1=3333.33，HC2=3333.33，HC3=3333.34）。
  - 减少HC与预算 ：
    - 减员方式 ：不针对HCM中已有的具体HC实例进行直接销毁或扣减。而是在目标部门（项目）下，新建一个特殊的**“减员”HC**。
    - 负数记录 ：该“减员”HC的编制数量及预算金额，在系统底层均以 负数 形式记录，从而在部门汇总层面实现HC总数和总预算的冲减。
- 场景B：不增加/减少HC，仅增加/减少预算
  
  - 特殊载体机制 ：因不动HC，为承载这部分纯预算的变动，HCM引入“人力外包-补差额HC”作为虚拟载体。
  - 初始化规则 ：当该部门第一次发生此类“只动预算不动HC”的业务落地时，系统将 自动为其生成 一个“补差额HC”。
  - 预算增减逻辑 ：OA传来的增加/减少的预算金额，直接在这个“补差额HC”的账户上进行加减。
  - 兜底穿透规则 ：若发生减少预算操作，且“补差额HC”当前的余额不足以扣减时， 允许其预算金额被扣减至负数 。
3. 异常处理与重试机制：

- 接口处理过程中若发生任何异常（如数据库锁、部门突然被撤销、网络超时等），HCM应阻断落地进程，并将具体的 失败原因及异常状态码同步返回给OA系统 。
- 由OA系统根据返回结果决定后续动作（如：系统自动发起重试、流程挂起等待人工干预等）。
