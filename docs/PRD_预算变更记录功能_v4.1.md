# HCM 系统：预算变更记录功能全量详细设计方案 (v4.1)

## 1. 方案背景与核心价值

### 1.1 业务背景
目前 HCM 系统的预算变更分布在审批、调动、组织调整等多个模块。为了实现“人财对账”的精准性，财务系统需要获取每一笔变更的明细数据（科目级 + 项目标签级）。本功能作为 HCM 系统与财务系统之间的“数据枢纽”，通过结构化的日志记录，追踪所有涉及编制（HC）与人力成本预算的业务变动。

### 1.2 核心价值
- **可视化追溯**：为 HR 提供直观的变更流水查询，数值增减带颜色区分，支持科目级下钻。
- **自动化集成**：通过数据中台准实时同步至财务系统（EPM/ERP），消除手工对账误差。
- **精细化管控**：确保所有预算变动均可下钻至 14 类标准科目及对应的项目标签，辅助经营决策。

## 2. 用户旅程与关键场景

### 2.1 典型用户旅程
1. **触发变更**：HR 在“人员调动”模块发起一个带 HC 的跨部门调动流程。
2. **流程审批**：相关领导审批通过，业务生效。
3. **日志落地**：系统后台自动拆解该笔业务涉及的 14 类科目明细，生成 `LOG-2026xxxx` 编号。
4. **查询回溯**：财务专员在“预算变更记录”页面，通过搜索“调动”或“部门名称”，查看到该笔变动，并展开明细确认科目分摊（如工资、公积金等）是否正确。
5. **中台同步**：数据中台每 2 小时自动抓取该记录，同步至财务系统。

### 2.2 关键业务场景
| 场景名称 | 业务逻辑 | 关键挑战 |
| :--- | :--- | :--- |
| **带 HC 调动** | 涉及调出部门的预算调减与调入部门的预算调增。 | 需确保调出与调入的两条对冲日志逻辑严密。 |
| **预算审批生效** | 针对特定部门的增量预算申请通过。 | 需将总额精确拆分至 14 个标准财务科目。 |
| **补差额静默处理** | 因入职时间差异或调薪导致的系统自动重算补差额。 | **不生成日志**，避免对外部财务系统造成噪声干扰。 |
| **部门撤销/剪切** | 组织架构调整导致的大规模预算平移。 | 涉及多条日志的批量生成与关联关系。 |

## 3. 前端交互与信息架构

### 3.1 界面原型与样式规范
- **组件选型**：使用 Ant Design Vue 4.x 的 Table 组件实现“嵌套表格”结构。
- **交互逻辑**：
    - **默认视图**：展示汇总列表，每行代表一次业务变更。
    - **展开交互**：点击行首的“+”号，在当前行下方展开“变更明细”子表。
- **视觉反馈**：
    - 数值增加：**红色**并带“+”前缀。
    - 数值减少：**绿色**并带“-”前缀。
    - 变更类型：使用不同颜色的**标签（Tag）**区分（如：调动-橙色、审批-蓝色）。

### 3.2 汇总列表字段 (Master)
| 字段名称 | 数据类型 | 说明 |
| :--- | :--- | :--- |
| 日志编号 | 字符串 | 全局唯一 ID，例：LOG-20260413-001 |
| 发生时间 | 日期时间 | 业务生效的精确时间（秒级） |
| 变更类型 | 标签 | 预算审批、带 HC 调动-调出/调入、部门撤销、部门剪切等 |
| 涉及部门 | 字符串 | 发生变更的目标部门全称及代码 |
| 类别 | 字符串 | 人员类别（如正编、实习生）或费用类别（如加班费） |
| HC 变动 | 整数/字符 | 编制数量增减，纯费用变更显示为“-” |
| 总额变动 | 货币 | 该笔业务涉及的预算总金额变动（元） |

### 3.3 嵌套明细子表字段 (Detail)
当用户展开记录时，展示该笔业务拆分后的财务科目分布：
| 字段名称 | 说明 | 示例 |
| :--- | :--- | :--- |
| 财务科目 ID | 财务系统中的标准科目编码 | ACC-001 |
| 财务科目名称 | 14 类标准科目之一 | 工资预算 |
| 项目标签 | 该科目金额归属的具体项目 | 完美世界项目 A |
| 变动金额 | 该科目在该项目下的具体变动值（元） | +150,000.00 |

## 4. 后台系统与数据模型

### 4.1 传输机制与格式
- **传输媒介**：通过数据中台（Data Mid-end）进行异步中转。
- **数据格式**：标准 JSON 格式。
- **同步频率**：HCM 实时落地日志表，中台每 2 小时抽取增量快照。

### 4.2 核心数据表结构
#### 主日志表 (budget_change_log)
| 字段名 | 类型 | 说明 | 约束 |
| :--- | :--- | :--- | :--- |
| log_id | String | 日志编号 (PK) | 主键 |
| biz_type | Enum | 业务类型枚举 | 非空 |
| dept_id | String | 涉及部门代码 | 外键 |
| person_category | String | 类别 | 非空 |
| hc_delta | Integer | HC 变动数 | - |
| amount_total | Decimal | 总变动金额 | - |
| is_supplement_diff| Boolean | **补差额标识（true 则前端不显示且不传输）** | 默认 false |
| effective_time | Datetime | 发生时间 | 默认系统时间 |

#### 明细表 (budget_change_detail)
| 字段名 | 类型 | 说明 | 约束 |
| :--- | :--- | :--- | :--- |
| detail_id | String | 明细唯一 ID (PK) | 主键 |
| log_id | String | 关联主日志 ID | 外键 |
| account_id | String | 财务科目 ID | 非空 |
| project_tag | String | 项目标签 | 可为空 |
| amount_delta | Decimal | 明细变动金额 | 和需等于主表总额 |

## 5. 预算科目全量清单 (14类)
所有明细数据必须覆盖以下标准科目：
1. 工资预算 | 2. 绩效工资 | 3. 商业保险 | 4. 工会经费 | 5. 饭补 | 6. 工伤保险 | 7. 医疗保险 | 8. 失业保险 | 9. 养老保险 | 10. 公积金 | 11. 签字费 | 12. 离职补偿金 | 13. 加班费 | 14. 其他人力成本预留

## 6. 验收标准与埋点

### 6.1 验收标准 (AC)
1. **前端展示**：数值颜色区分正确，展开后明细清晰且总额对齐。
2. **后台逻辑**：`is_supplement_diff` 过滤逻辑生效，中台接收到的 `items` 数组完整。
3. **一致性**：财务系统接收的总额与 HCM 审批单据总额完全一致。

### 6.2 埋点需求
- `log_view`: 记录查看 | `log_expand`: 明细下钻 | `log_export`: 数据导出

## 7. 风险评估与迁移计划
- **风险**：数据不一致。**对策**：增加“日终对账脚本”比对余额表与日志表。
- **性能**：单表突破百万级。**对策**：按年度分区存储并建立复合索引。
- **迁移**：存量数据 v3.0 前无明细。**对策**：历史数据标记“无明细”或按默认比例模拟拆分。
