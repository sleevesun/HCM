【PRD】HCM_部门级预算申请与移动端
审批界面优化
1. 需求背景与目标
1.1 需求背景
1. 预算录入痛点：业务部门在编制“加班费”、“离职补偿金”、“外派补贴”等公
共/部门级预算时，缺乏整体视角的录入与查看工具。
2. 移动端审批体验痛点：当前移动端预算审批卡片数据展示格式不够直观，信息排版
拥挤，无法直观的查看各类人员的预算变化
1.2 业务目标
1. 上线「部门级预算申请」（PC端）：支持基于组织架构树的统筹录入，实现申请-
审批-使用-比对的闭环管理
2. 优化「移动端审批界面」（移动端）：落地移动端“渐进展开”的方案，采用“申
请前 → 申请后”的直观对比展示，强化增减颜色语义，通过无变化数据折叠实现
视觉降噪。
2. 用户角色与使用场景

预算填报人 (主要为 在 PC 端预算调整页面，统筹录入下属 能够直观看到部门层级关系；能快速录

3. 功能需求详述：部门级预算申请（PC 端）
3.1 入口与形态
 入口：PC端预算调整页 -> 点击【部门级预算变更】按钮。
 交互形态：居中弹窗，主体为 Tree-Table（树形表格）。

3.2 核心字段与展示规则
1. 表头设计：
¢ 采用两级表头。
¢ 一级表头：加班费、离职补偿金（注：外派补贴预计在填报2027年预算时
新增为部门级预算，本期暂不实现）。
¢ 二级表头：申请前(元)、申请后(元)、变化(元)。
2. 部门架构树：
¢ 首列为当前所选部门名称，默认展开一层。
¢ 组织树默认仅展示到04级部门，05级及以下部门不展示
3. 数据联动与编辑：
¢ 申请前：只读，千分位展示。
¢ 申请后：各个部门节点均可编辑（使用 InputNumber，支持千分位录入）。
“部门合计”行不可编辑。
¢ 自动汇总：合计行的“申请后”数据，实时等于其所有当前页面所有部门节点
之和。
¢ 变化值计算：变化 = 申请后 - 申请前。带正负号（如 +5,000 或 -2,000）。

4. 视觉降噪：
¢ 变化值 > 0 时，文字标红。
¢ 变化值 < 0 时，文字标绿。
¢ 变化值 = 0 时，文字标灰（0不带符号）。
3.3 生成申请后数据
 触发时机：当用户保存部门级预算数据时，自动生成包含部门级预算和所有HC在
内的整体申请后数据
 HC数据计算逻辑：自动按照当前数据“不变”进行处理
3.4 部门级余额动态平摊（远期规划，本期暂不实现）
注意：当前系统（HCM）仍保持原有逻辑，即直接存取一个全年的数据总包，不做
按月平摊控制。以下“余额动态平摊”逻辑为后续优化建议，本期暂不实现。
部门级预算（加班费、离职补偿金、外派补贴）不直接由用户按月录入，而是用户录入
全年总额，系统按以下规则自动在月份间动态平摊：
 核心原则：全年的预算总盘不因实际使用的超支或结余而改变。
 分摊规则：未来的月份分摊所有剩余预算额度；已发生月份的预算 = 实际发生值。
当到达第12个月时，12月份的预算 = 剩余的全部额度（不再根据12月的实际
值去 true up 预算总盘）。
 场景举例：
¢ 假设 2026 年编制 2027 年的预算，全年离职补偿金录入为 120万。
¢ 期初状态：120万平摊到12个月，每月预算为 10万。
¢ T1时刻（2027年2月初）：1月实际发生离职补偿金为 0万。则1月预算
更新为 0万，剩余的 120万 重新平摊到 2-12 月（11个月），每月预算
更新为约 10.9万。
¢ T2时刻（2027年3月初）：2月实际发生离职补偿金为 60万。则2月预
算更新为 60万。全年剩余预算 =120 -0(1月) - 60(2月) = 60万。这60
万将重新平摊到 3-12 月（10个月），每月预算更新为 6万。

¢ T3时刻（2027年4月初）：3月实际发生离职补偿金为 70万（超出了剩
余的60万）。则3月预算更新为 60万（或者按实际记录70万，但剩余月
份预算归0），全年预算总盘保持120万不变，4-12月的分摊额度为 0万。
¢ T12时刻（2027年12月）：前11个月结算完毕后，12月份的预算直接
等于剩余的全部可用额度。即使12月实际发生额大于或小于该剩余额度，也
不会再反向去修改全年的预算总盘（no true up）。
 触发时机：每个月11日根据上个月实际发生数据同步时，系统需触发重新计算逻
辑。
4. 功能需求详述：移动端审批界面优化
4.1 整体布局
页面分为多个主要的卡片组，自上而下依次为：
1. 【部门整体】：核心全局指标。
2. 【人员工薪预算变化】：按用工类型拆分的人员相关预算。
3. 【部门级预算变化】：非挂载到具体人的公共预算。
4. 【审批附加信息】：包含申请说明、审批记录与审批意见模块。
方案整体视图（未展开）：

4.2 【部门整体】卡片
 指标精简：仅保留最核心的“总工薪预算(万)”。
 格式调整：摒弃 申请后(+变化) 的形式，改为 申请前数值 → 申请后数值 的演
进式表达（例如：2,214.0 → 2,502.0）。

 颜色规则：箭头 → 后面的数值，若大于申请前则标红，若小于申请前则标绿，等
于则标灰。
4.3 【人员工薪预算变化】卡片组（渐进式手风琴）
按用工类型（正编、人力外包、实习、劳务派遣、兼职等）独立成卡片。动态展示规则：
若某类人员在当前部门/申请中不存在（即前后数据均为无），则不展示该类别的卡片。
4.3.1 卡片缩略态（未展开）
采用 3 行布局：
 行1（标题）：左上角为人员类型（如“正编”），右上角为展开控制图标 ⌄。
 行2（指标1）：左侧文字 HC，右侧数据展示为 申请前 → 申请后（例：50→
54）。
 行3（指标2）：左侧文字 年度工薪(万)，右侧数据展示为 申请前 → 申请后（例：
1,800.0 → 1,980.0）。
 颜色规则同部门整体，箭头后数值根据增减变色。
4.3.2 无变化状态处理（视觉降噪）
若某类人员的 HC 与工薪变化均为 0（例如：实习生申请前24万，申请后24万）：
 交互限制：该卡片不可点击展开详情。
 视觉呈现：
¢ 行1：右上角原本的 ⌄ 替换为灰色的“无变化”标签。
¢ 行2/行3：不再展示 申请前 → ，仅靠右展示当前的“申请后”数值（如 5），
不再额外显示“无变化”文字。
4.3.3 卡片详情态（展开后）
用户点击可展开的卡片后，纵向平铺展示详细指标：
 展示指标：HC、月度工薪(万)、年度工薪(万)。
 指标结构：每个指标拆分为三行进行严谨对比：“申请前”、“申请后”、“变化”。
 颜色规则：“变化”行的数值，增加显示红色，减少显示绿色，0显示灰色。

4.5 【审批附加信息】模块

1. 申请说明：展示用户发起申请时填写的具体变更内容与原因。
2. 审批记录与标准路径：
  ¢ 采用纵向时间轴形式展示前序及后续审批节点。
  ¢ 展示字段：每个节点展示 审批人、审批结果（如已同意、待审批等）、审批
  节点名称、审批时间、审批意见。
  ¢ **后台逻辑提示**：系统需根据部门所属板块自动渲染对应的预设路径：
    - **游戏 & 电竞**：HRD -> C&B -> 二级部门负责人 -> HRBP Head -> 业务HR Head -> 1.1部门负责人(如有) -> CEO & 集团HR。
    - **集团职能 & 影视**：HRD -> C&B -> 二级部门负责人 -> 业务HR负责人 -> 1.1部门负责人(如有) -> 集团HR -> 董事长。
3. 审批意见：
¢ 供当前审批人填写的文本域
¢ 必填校验：当用户点击底部“拒绝（驳回）”按钮时，校验意见是否为空；若
空则 Toast 提示填写，界面上不显示红星。点击“通过”时不强制校验。
5. 业务规则与约束
1. 预算周期：当前前端界面展示的数据均为“全年预算”（Annual Budget）视角。
2. 数据精度：
¢ PC端金额（部门级预算申请）：统一精确到个位（元）。
¢ 移动端总览与年度金额：统一精确到小数点后 1 位（万）。
3. 兜底策略：所有卡片渲染均基于后端下发的数据结构，前端不做硬编码过滤。只要
后端返回了某类人员且该人员有预算数据，前端即按规则渲染（支持未来可能新增
的用工类型）。
