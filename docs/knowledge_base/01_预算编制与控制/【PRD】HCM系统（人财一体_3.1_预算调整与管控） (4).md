【PRD】HCM 系统（人财一体_3.1_预算
调整与管控）
一、 背景和目标
1.1 背景
HCM3.0 上线后，解决了预算编制的线上化问题，但仍缺少关键的执行管控能力（HCP
2.1已实现功能）：
 缺少执行校验：入职、调动流程无法自动校验 HC 和预算，存在超编风险。
 缺少业务灵活性：无法支持紧急的“过渡期 HC”替换，或在部门间调剂预算。
1.2 目标
本次升级的核心目标是实现 HC 及预算的线上化闭环管控：
 在入职、调动等流程中加入实时校验，确保“无预算，不发生”。
 上线“过渡期 HC”功能，提升资源调配效率。
 将 HC 申请、调整、使用、释放全流程线上化，做到过程可追溯。

二、 新增功能需求
2.1 HC 及预算管控
2.1.1 核心设计原则
所有（非例外的）人员入职和调动，都必须通过本功能的实时校验。系统将作为守门员，
为PS系统提供实时的“准入”或“拒绝”判断。

2.1.2 管控对象与场景
 管控对象：HC 数量、工资预算（注意：仅管控工资，不管控工薪）。
 管控场景：PS系统中的“入职申请”和“（不带HC）调动申请”。
2.1.3 核心产品逻辑
1. 系统交互逻辑
 触发时机：当HR在PS系统发起“入职”或“（不带HC）调动”流程，点击“提
交”时。
 系统动作：PS系统向HCM系统发起一次实时的“预算可用性校验”请求。
 校验关键信息：该请求必须包含发起校验所必需的业务信息，包括：
¢ 目标部门（二级管控部门）
¢ 员工类型（正编：正式员工+试用员工、实习）
¢ 预计入职/调入日期
¢ 员工月薪
 HCM反馈：HCM系统在实时计算后，向PS系统返回明确的“校验通过”或“校
验拒绝”的结果。
 PS端处理：当PS系统收到“校验拒绝”的结果时，必须阻止HR提交该流程，
并向HR弹出HCM系统返回的拒绝原因（例如：“2026年Q1正编HC余额不
足，无法发起流程。”）。
2. 动态可用额度计算
系统在收到校验请求时，实时计算“可用HC”和“可用预算”。
 可用 HC 计算公式：
可用HC=[部门总HC]-[在职人数]-[在途入职（HIR）]-[在途不带HC调入（XFR）]
+ [在途不带HC调出] + [在途离职（TER）]（已审批完成）
在职人数：与初始化生成HC时逻辑保持一致。排除虚拟公司员工（小号），排除NOPAY
的正编人员。
 可用预算计算逻辑：
系统需根据 员工类型 和 入职/调入日期 确定校验规则。
¢ 规则A：正编

 HC校验：检查 入职/调入日期 所在季度末（3/31,6/30,9/30,12/31）
的 可用HC 是否 > 0。
 预算校验：检查 入职/调入日期 所在季度末的 可用工资预算 是否 > 员
工月薪。
¢ 规则B：实习生
 HC校验：不校验 HC 数量。
 预算校验：检查该部门全年的 [实习生总预算] - [实习生已用预算] - [实
习生在途预算] 是否 > (实习生月薪 * 剩余月数)。
 预算校验：检查该部门本月度的 [实习生总预算] - [实习生已用预算] -
[实习生在途入职预算]+实习生在途离职预算 是否 > 实习生月薪
 简单来说：月薪小于月度余额，可以入职
3. “在途”状态处理（核心闭环）
为防止并发申请导致预算超发，系统必须管理“在途”状态。
 占用（Hold）：当HCM系统“校验通过”一个请求后，必须立刻为该部门的 [在
途入职] 或 [在途调入] 数量+1，并（在计算后续校验时）扣减相应的 [在途预算]。
 释放（Release）：如果该笔流程在PS系统中被**“驳回”或“撤销”，PS系
统必须**通知HCM系统。HCM系统收到通知后，将对应“在途”占用-1，并释
放预算。

2.2 HC 及预算释放
2.2.1 核心设计原则

员工离职或调离，其占用的HC和预算编制必须自动释放回部门的资源池，供后续招聘
使用。此过程必须由系统自动触发，无需人工干预。
2.2.2 系统交互与产品逻辑
 触发时机：当PS系统中的“离职流程”或“（不带HC）调动流程”审批通过，
状态变为“待生效”时。
 系统动作：PS系统在流程审批通过后，必须向HCM系统推送事件通知。
 通知关键信息：该通知必须包含：
¢ 员工
¢ 调出的二级管控部门
¢ 预计离职/调出日期
¢ 事件类型（如：离职、不带HC调动）
 HCM核心逻辑：
a. HCM收到事件后，根据员工信息找到该员工占用的HC。
b. 系统将该HC的状态更新为“在途离职/调出”。
c. 释放逻辑：系统以 预计离职/调出日期 为界：
 预计离职/调出日期 所在月份之前（含当月）的HC和预算保持“占用”
状态。
 预计离职/调出日期 所在月份的当月开始，该HC和预算被释放，部门的
可用HC 和 可用预算 额度增加。
 示例：员工3月15日离职，则该HC在3月仍计为占用，从4月1日
起，该HC和预算释放回部门资源池。
2.2.3 异常处理：离职/调动撤销
 场景：员工的离职/调动流程在“待生效”状态下被撤销。
 系统动作：PS系统必须发送一个“撤销”事件通知HCM系统。
 HCM核心逻辑：HCM收到事件后，将对应的“在途离职/调出”状态取消，HC
和预算恢复为“占用”状态，释放逻辑被中止。

2.3 组织架构调整下的预算联动
2.3.1 核心设计原则
预算数据必须与组织架构强绑定。当组织架构发生变动时，预算（尤其是空HC和部门
预算）必须随之自动、准确地转移，确保数据一致性，避免数据丢失或归属错乱。
2.3.2 场景一：部门整体平移（剪切）
 业务场景：在「组织树1.0」中将一个现有部门（如“美术中心”）从“A部门”
剪切并粘贴到“B部门”下。
 系统交互：「组织树1.0」系统在完成保存后，推送“组织节点移动”事件。
 HCM核心逻辑：
a. 识别：在此场景下，部门的唯一ID 没有发生变化（例如，美术中心的ID仍为
'10086'）。
b. 处理：HCM系统无需执行任何预算数据转移操作。由于HCM中所有预算数
据（HC、部门预算、历史数据）均已关联到该部门ID，当该部门在组织树上
的父节点变更时，预算数据的归属自动随之变更。
2.3.3 场景二：部门撤销新建（承接）
 业务场景：撤销原部门“原美术中心 (ID: 001)”（及其下属所有子部门），其未
来的空HC预算及部门预算，需要被“新美术中心 (ID: 002)”（及其新建的子部
门）承接。
 「组织树1.0」系统改造：
¢ 触发点（申明）：在「组织树1.0」的“新建部门”功能界面（以及“修改部
门”界面），增加以下可选字段：
i. “承接预算自（原部门ID）”（一个部门输入框，输入ID带出部门名称）
ii. “预算承接生效月份”（一个月份选择器，例如 "2026-01"）
¢ 交互逻辑：SSC在新建“新美术中心
¢ (002)”时，就在此字段指定承接自“原美术中心 (001)”。（同理，新建“新
概念 (102)”时，指定承接自“原概念 (101)”）。

 系统交互（数据传递与存储）：
b. 组织树 -> PS：「组织树1.0」在“新建/修改”部门保存后，必须将这个映
射关系 (新部门ID, 原部门ID, 生效月份) 传递给 PeopleSoft(PS) 系统。
c. PS 存储：PS系统收到后，必须将此映射关系存储在一个“预算承接待办表
(Queue)”中，状态为“待执行”。
 HCM/PS 核心逻辑（延迟执行）：
a. PS 轮询作业：PS系统必须建立一个每日定时作业（Job），轮询这张“待办
表”。
b. 作业校验逻辑：对于表中的每一条“待执行”任务，PS作业必须实时校验：
 条件A：原部门ID 在「组织树1.0」中的状态是否已变为“失效”？
 条件B：该条任务的“预算转移状态”是否为“未执行”？
c. 触发转移：当 条件A 和 条件B 同时满足时，PS系统立即调用HCM系统
的预算转移接口。
d. HCM 接口：HCM系统提供一个接口，接收 原部门ID, 新部门ID, 生效月
份。
 HCM 预算转移逻辑（与原方案一致）：
a. HCM收到PS的转移指令后，立即触发预算转移程序。
b. 定义转移时间点：以 预算承接生效月份 为转移的起始点。
c. 转移未来数据：系统查询 原部门ID 名下所有自 生效月份 起（含当月）的
空HC及其预算，以及部门级预算余额，将这些记录的归属部门批量修改为 新
部门ID。
d. 保留历史数据：所有 原部门ID 在 生效月份之前的已占用HC预算和已发生
历史数据将永久保留在 原部门ID 下，且向上汇总到其上级部门的历史数据当
中
e. 状态回写：HCM执行成功后，必须通知PS系统，PS将“待办表”中该条任
务的状态更新为“已完成”，防止重复执行。
 异常处理：
¢ 如果HR新建部门时未填写“承接预算部门ID”，则HCM不执行任何预算转
移操作，"002" 将是一个全新的、预算为空的部门。

2.3.4 部门撤销时各类型预算转移承接逻辑
1. 基础定义
 非实体部门：指层级属性为 x.1级、02级、01级 的组织节点。此类部门不可承
接具体的 HC 编制预算（因为此类部门不直接承载人员）。
 实体部门：除上述以外的其他部门节点。
 默认继承规则：在“直接撤销”场景下，默认继承方恒定为 直接上级部门。
2. 数据范围与递归原则
 递归处置：如果被撤销部门包含下级子部门，系统需遍历其所有下级分支：
¢ 将所有下级部门视为“待处置单元”。
¢ 按照本规范逻辑，顺着组织树向上寻找最近的有效上级进行归集。
 人员状态：部门撤销的前提是该节点下（含子部门）不存在在职人员。
3. 场景一：部门直接撤销
定义：部门裁撤，业务终止。默认接收方：上级部门。

1. 部门级预算
包含加班费、离职补偿金、签约金等科目。
 处置动作：全额上移。
 接收方：上级部门。
 逻辑规则：此处不校验上级部门的实体/非实体属性，强制兜底。
¢ 已使用金额（历史）：
 动作：全额归属上级。
 公式：上级接收的已发生额 = 原部门实际已发生金额。
¢ 预算：
 动作：全额划转上级。
 公式：上级接收的预算 = 原部门预算。
¢ 汇总结果：上级部门的已使用金额和总预算从观感上没有变化（撤销前下级部
门的已使用和预算也会汇总到上级，现在直接属于上级）。
2. HC 级预算（空HC）
包含空 HC 对应的工资、饭补、社保等相关预算。
 切分逻辑：按时间维度切分为“历史”与“未来”。
¢ 历史预算
 定义：撤销生效月之前的预算数据。
 处置动作：上移。
 接收方：上级部门。
 逻辑：历史沉没成本回归上级，不进行属性校验。
¢ 未来预算
 定义：撤销生效月及之后的预算数据。
 处置动作：条件上移 或 作废。
 判定逻辑：检查 上级部门 的属性。
 IF 上级 = 实体部门：
¢ 动作：上移。空 HC 及未来预算额度转移给上级。
 IF 上级 = 非实体部门 (x.1, 02, 01级)：

¢ 动作：作废。
¢ 结果：空 HC 关闭，未来预算额度直接清零。
3. 补差额 HC 级预算
 历史预算：数据合并至上级部门（不校验属性）。
 未来预算：
¢ 第一步：校验部门是否实体属性
 IF 上级 = 非实体部门：未来的预算作废，原补差额 HC 关闭。
 IF 上级 = 实体部门：进入第二步“数据合并”。
¢ 第二步：数据合并
 将下级部门的补差额预算合并到上级部门
 如果上级部门当前无补差额HC，则先创建补差额预算后再转移预算
4. 场景二：原部门撤销，指定新部门继承
定义：部门裁撤，但业务职责划转给指定的“继承部门”。涉及方：上级部门 (承担历
史)、继承部门 (承接未来)。
4. 部门级预算
 预算余额
¢ 处置动作：划转给继承部门。
¢ 范围：包含该部门及其所有下级部门的当前剩余可用预算。
¢ 公式：继承部门接收额 = 原部门(含下级)当前预算余额。
¢ 特殊：当余额≤0时，则转移的余额为0（不转移负数）。
 历史预算
¢ 处置动作：归属原上级部门。
¢ 逻辑推导：
 系统执行公式：上级接收历史预算额 = 原部门(含下级)全年总预算 - 转
移到继承部门的预算。
 C. 已用部分
¢ 处置动作：归属原上级部门。

¢ 逻辑推导：
 系统执行公式：上级接收额 = 原部门实际已发生金额。
5. HC 级预算（空HC）
 历史预算
¢ 定义：撤销生效月之前的预算数据。
¢ 处置动作：上移至原上级。
¢ 逻辑：继承部门只对未来负责，不对该部门过去的历史成本负责。
 未来预算
¢ 定义：撤销生效月及之后的预算数据。
¢ 处置动作：条件划转 或 作废。
¢ 判定逻辑：检查 继承部门 的属性。
 IF 继承部门 = 实体部门：
 动作：划转。空 HC 及未来预算转移给继承部门。
 IF 继承部门 = 非实体部门：
 动作：作废。
 结果：继承方虽接业务但不接 HC，该 HC 直接关闭，预算回收。
6. 补差额 HC 级预算
 历史预算：数据合并至上级部门（不校验属性）。
 未来预算：
¢ 第一步：校验继承部门是否实体属性
 IF 上级 = 非实体部门：未来的预算作废，原补差额 HC 关闭。
 IF 上级 = 实体部门：进入第二步“数据合并”。
¢ 第二步：数据合并
 将撤销部门的补差额预算合并到指定继承部门
 如果继承部门当前无补差额HC，则先创建补差额预算后再转移预算

2.4 人员调动下的预算自动调整
2.4.1 核心设计原则
支持“HC随人走”的业务需求，当发生“带HC调动”时，调入部门无需预留HC，
系统自动将HC和预算从调出方划转至调入方。
2.4.2 场景一：标准“带 HC 调动”（PS 流程）
 业务场景：HR在PS系统中为员工（如“张三”）发起“调动”流程，并明确勾

¢ 原部门
¢ 新部门
¢ 生效日期
¢ 调动类型（“带HC”）
 HCM核心逻辑：
f. HCM收到事件后，根据员工信息找到该员工（张三）占用的HC。
g. 系统以 生效日期 为界：

 从 生效日期 所在月份开始，系统自动将该HC和预算记录的 归属部门

h. 无需校验：此操作无需校验新部门 是否有可用HC，因为HC是随人带入的。
i. 历史数据保留：张三在 原部门 已发生的历史成本，将永久保留在 原部门。

 功能界面与逻辑：
j. 界面：与现在调用界面相同。
k. “执行”按钮逻辑：

2.5 过渡期 HC 申请功能（正编）
2.5.1 核心设计原则
为业务部门提供一种临时的、有时间限制的HC，以应对紧急的人员替换需求（如员工
即将离职，需提前招聘）。此HC不增加年度预算，但增加HC数量，且其工资受参考
人限制。

1. 申请界面（参照原型）

 规则1（薪资上限）：系统自动获取 [被替换人] 的当前月薪，作为该过渡期HC
的 [工资预算上限]。后续入职的人员工资不得超过此上限。
 规则2（时长限制）：[HC有效截止日期] - [申请日期] 不得超过5个月。
 规则3（季度末限制）：[HC有效截止日期]仅提供季度末（3、6、9、12月）选

¢ 系统实时计算：[部门正编HC编制总数] * 2.5% (向上取整)
¢ 校验：[过渡期HC总数] 不得超过 [2.5%上限]。如超过，则提示（如UI原

 简化审批：申请提交后，进入简化审批流（仅需 HRD 和 业务部门负责人 审批）。

 系统动作：
n. 扫描所有已生效的过渡期HC。
o. 如果 [当前日期] > [HC有效截止日期]：
 如果HC为空：系统自动将该HC设为“已失效”，可用HC 数量-1。
 如果HC已占用（即已有人入职，但部门没有对应的人员离职）：系统自

2.6 “加班费”与“离职补偿金”分级预警管控详细方案
2.6.1 核心设计原则
1. 统一适用：本方案的逻辑、状态管理和通知模板，将统一应用于“加班费”和“离
职补偿金”两个预算项目。
2. 每月通知： 通知逻辑不设防重状态。系统在每月的固定检查点（11号）均会进行
判断，无论上月是否通知，本月只要满足条件即再次发送通知。

3. 最高档位优先： 当消耗比例同时满足多个档位时（如95%同时满
足>90%, >70%, >50%），系统仅发送当前命中的最高档位（90%）的通知。
2.6.2 核心产品逻辑与触发机制
 触发机制（代替事件A、B）：
¢ 触发时机： 系统不再实时触发，改为每月11号（或约定日期，即收到上月
发薪实际数据后）执行一次定时校验任务。
 核心产品逻辑：
a. 每月检查： 每月11号，任务启动，遍历所有部门的“加班费”和“离职补偿
金”项目。
b. 实时计算： 计算[当前消耗百分比] = [累计已消耗金额] / [年度总预算]。
 系统按倒序（从100%到50%）检查消耗百分比。
 一旦命中一个档位，立即发送该档位的通知，并停止对该部门该项目的后
续检查。
 规则1 (100% 检查): 如果 [当前消耗百分比] > 100%，触发100%预
警通知。（检查停止）
 规则2 (90% 检查): 如果 [当前消耗百分比] > 90% (且 <= 100%)，
触发90%预警通知。（检查停止）
 规则3 (70% 检查): 如果 [当前消耗百分比] > 70% (且 <= 90%)，触
发70%预警通知。（检查停止）
 规则4 (50% 检查): 如果 [当前消耗百分比] > 50% (且 <= 70%)，触
发50%预警通知。（检查停止）
 每月重复通知（无防重）：如果某部门9月消耗95%（收到90%通知），10月
消耗96%（未追加预算），则10月11号检查时，依然会命中规则2，再次发送
90%通知。

2.7.2 预算及 HC 变动日志
构建“预算流水账”能力，实现预算与编制的变更时间完整记录。无论是人工操作还是
系统自动触发（如组织搬迁、人员调动）导致的额度变化，都必须生成不可篡改的流水
记录，解决业务部门对“预算去向”的查询需求。
1. 数据记录范围
系统需捕获以下所有导致部门预算或HC发生变化的事件，并写入日志表：
a. 人员异动类：入职占用、离职释放、带HC调入/调出。
b. 组织变更类：部门平移导致的预算归属变更、部门新建时的预算承接。

i. 该类预算变更，因预算的主体部门没有发生预算和HC的变化，因此该记

2.9 调薪场景-预算校验
为防止因调薪导致的成本超支，系统需在调薪单据发起阶段引入“强管控”机制。通过
与 HCM 预算管理模块对接，校验调薪生效日期对应季度末的工资预算余额，并对拟
增加的人力成本进行预占用。
 预算校验日期判定（季度末规则）
系统根据单据中选择的 “生效日期”，自动推算需请求的预算余额截止月份。
¢ 计算公式：
 生效日期月份 ∈ [1, 2, 3] → 校验 3月 预算余额
 生效日期月份 ∈ [4, 5, 6] → 校验 6月 预算余额
 生效日期月份 ∈ [7, 8, 9] → 校验 9月 预算余额
 生效日期月份 ∈ [10, 11, 12] → 校验 12月 预算余额
 校验与拦截规则
系统需对比“本次拟调薪增加额”与“可用工资预算余额”。
¢ 计算本次增加额：
 增加额度 = Σ (员工调整后月薪 - 员工调整前月薪)

 需按员工所属的预算管控部门分别汇总。
¢ 通过标准：
 所有涉及的预算管控部门满足：可用预算余额 ≥ 本次拟调薪增加额
¢ 拦截逻辑：
 若 任意一个 涉及的预算管控部门（或其下属部门）余额不足（即 增加
额度 > 余额），则 阻断单据提交。
 错误提示：弹窗提示具体的超支部门名称及超支金额（例如：“【美术部】
2025年9月预算余额不足，无法提交”）。
 预算占用/释放
本调薪流程将作为“在途单据”纳入统一预算管控体系。当其他业务（如入离职、调动）
查询预算时，系统会实时计算本流程中“拟调整但未生效”的金额，作为预占用的支出。
¢ 预算占用的计算规则
 涨薪场景（成本增加）：
 逻辑：单据要和其他单据一起提供给HCM，系统需预留这部分钱，
防止被其他业务（如正在进行的招聘）提前使用。
 占用额 =Σ (调整后月薪 - 调整前月薪)
 降薪场景（成本减少）：
 逻辑：当增加额度≤0，该流程信息不传输给HCM。既不释放原有的
预算额度。防止流程中途被驳回，而释放出的额度已经被其他业务花
掉，导致死锁。
 占用额 = 0（既不增加占用，也不释放余额）。
¢ HCM的响应机制
 当 HCM 请求“调薪模块当前占用了多少钱？”时，本模块需按 部门 和
月份 汇总上述计算出的在途占用金额，反馈给 HCM，确保公司整体预算
视图的准确性。

3.2 功能优化：非 PS 发薪人员（海外、派遣、兼职）实际数
据闭环

3.3.1 核心设计原则
打通“预算-实际”数据的最后屏障。将原先在线下支付、无法追踪的非PS发薪人员
（海外、派遣、兼职）的实际成本，通过数据中台汇集后，回流至HCM系统，形成完
整的部门实际支出视图。
3.3.2 核心产品逻辑
1. 数据归集（数据中台）
 数据源：SSC/海外员工管理员需按月将“海外”、“派遣”、“兼职”三类人员
的实际发薪结果（包含人员、部门、费用项目、金额）导入PS系统的指定模块。
 中台处理：数据中台T+1自动抓取PS中的发薪结果数据。
 中台聚合：数据中台将明细数据按[部门ID]、[预算年度]、[月份]、[预算项目]（如：
海外工资、海外保险、派遣费用等）进行汇总，形成部门级的月度实际支出。
 数据推送：数据中台将汇总后的实际支出数据，推送至HCM系统，HCM将其记
为对应部门的“实际已发生”金额。
2. 特殊逻辑：海外人员保险的“预算自动补增”
 业务场景：年度预算编制时，仅编制了海外人员的“工资”预算，未编制“保险”
等福利预算。但实际发薪时，产生了“保险”的实际支出。
 问题：这会导致HCM中该部门的“海外保险”项目出现 预算:0、实际:5000 的
情况，形成虚假的“超预算”。
 解决方案（预算自动对齐）：
¢ 触发时机：当HCM系统收到来自数据中台的，关于“海外人员保险”项目的
实际支出数据时。
¢ 系统动作：HCM系统在记录这笔“实际支出”的同时，必须自动为该部门、
该月份的“海外人员保险”项目增加一笔同等金额的预算。
¢ 示例：收到“A部门-5月-海外保险-实际支出:5000”，系统自动执行 A部
门-5月-海外保险-预算: +5000。
¢ 结果：该项目的 预算: 5000、实际: 5000，使用率100%，真实反映了业务
（未编预算，实报实销）。

¢ 审计：这笔系统自动增加的预算，必须在部门的预算调整日志中清晰可查，标
记为“系统自动补增：海外保险实际成本对齐”。
3.3 功能优化：TA 系统预算数据一致性校验（含后台界面）
3.3.1 核心设计原则
为确保HCM（人力预算）与TA（财务总账）两大系统的预算总额保持一致，需建立一
套自动化的数据对账机制，并为HR预算用户提供一个可视化的“对账驾驶舱”界面，
以主动发现和定位差异。
3.3.2 核心产品逻辑
1. 后台数据对账（数据中台）
 触发时机：每日凌晨，由数据中台自动执行。
 对账范围：二级部门及游戏工作室。
 DMP处理逻辑：
a. 从HCM系统获取[部门ID]及其[年度工薪预算总额]。
b. 从TA系统获取对应的[部门ID]及其[年度工薪预算总额]。
c. 从TA系统获取其“数据最后更新日期”（TA系统需提供此信息）。
d. 按部门逐一比对 [HCM预算] 与 [TA预算]，计算[差异金额]。
e. 将比对结果（部门、HCM金额、TA金额、差异金额、TA数据日期）写入一
张“对账结果表”，供HCM前端界面调用。
2. 前台可视化界面：“TA-HCM 预算一致性校验”
 功能定位：HCM后台管理功能
 界面核心元素：
