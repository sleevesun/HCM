【PRD】HCM_年初标识&管控校验&按钮
优化（2 月）
需求一：预算审批单“年初预算”打标
1. 业务场景
为了配合下游财务系统（TA）的预算差异计算，HCM 系统需要准确界定各部门每年的
“初始预算基准”，并作为独立的数据资产输送给数据中台。对于不直接发起审批的虚
拟管理节点，需支持数据的自动向上汇总。
2. 系统逻辑规则
2.1 自动打标逻辑（实体部门）
 触发节点：预算审批单的最终审批节点通过，且单据状态变更为“已生效”时触发。
 判断规则：系统按【预算年度】+【预算管控部门】进行校验。若该部门在该年度
下不存在任何带有“年初预算”标签的已生效单据，则将当前这第一笔生效单据自
动标记为“年初预算”。
 UI 展示：在预算单据列表页、单据详情页的单据编号旁，增加醒目的【年初预算】
Tag 标签。
2.2 虚拟部门的基准汇总逻辑
 虚拟部门定义：针对系统组织树中不直接发起、也不独立承载预算审批单的虚拟管
理节点。通过【部门层级】字段及特定 ID 进行精准识别，命中以下任一条件即判
定为虚拟节点：
a. 部门【层级】属性为 “01级” 或 “1.1级”。
b. 特殊业务架构节点：“游戏工作室群”（固定识别 ID：0106762）。
 计算规则（向上汇总）：虚拟节点的“年初预算”总额 = ∑（当前其所有向下直
属实体层级部门的“年初预算”总额）。

 动态更新
¢ 由于下属各实体部门的预算审批生效时间存在差异，虚拟节点的“年初预算”
数值呈现动态变更的情况。
¢ 每当其任一下级部门的“年初预算”被打标或发生金额修正并落地时，系统自
动触发一次向上的汇总计算，刷新该虚拟节点的年初预算汇总金额。
¢ 其他说明：系统允许并接受虚拟节点数据的“按天持续累加变多”状态。无需
等待其下属所有实体部门全部生效，中台及下游 TA 系统可随时抓取该虚拟节
点当前已汇总的最新快照数据。
2.3 人工干预机制（开发配置）
 由于错打标情况属于低频异常场景，为节省研发资源并收口高危操作权限，系统不
提供前台的干预按钮功能。
 操作方式（开发配置）：若确需修改（如调整单据 A 的标签并赋给单据 B），由
业务用户提交ISDO工单，技术人员通过后台直接执行 SQL 脚本或调用内部配置
接口进行状态修正。
 数据联动保障：后台配置修改生效后，系统必须支持触发一次对关联虚拟节点的基
准重算，并同步更新年初预算资产表。
2.4 数据落盘与中台交互
 独立建表：系统不再将此标识混在日常明细接口中传输，而是新建一张独立的表，
该表需同时存储实体部门与虚拟节点的数据行。
 中台读取：数据中台通过定时任务读取此资产，即可获取全集团的年初基准数据，
转交给 TA 系统。
需求二：预算管控部门层级校验功能
1. 业务场景
防止在调整组织架构或配置预算归属时，出现预算管控部门错位导致的逻辑问题，从而
引发预算数据出错。

2. 规则定义算法
在遍历组织树时，针对任何一个被勾选了“预算管控”属性的部门节点，执行以下三个
逻辑判断：
 规则 ①（二级找一级）
¢ 定位到任意【二级管控部门】节点，向上寻找一层（+1 级）。
¢ 该父节点必须是【一级管控部门】。
¢ 如果该二级部门自身同时也被勾选了【一级管控部门】，则本条规则直接视为
通过。
 规则 ②（防重叠校验）
¢ 定位到任意【二级管控部门】节点，从该节点向根节点溯源遍历整条汇报线；
¢ 在此路径中，不允许出现第二个被勾选了【二级管控部门】的节点；
¢ 同时，该路径上也只能有一个被勾选了【一级管控部门】的节点（可以是该部
门自身）。
 规则 ③（一级找二级）
¢ 定位到任意【一级管控部门】节点，向下寻找一层（-1 级所有直属子节点）。
¢ 其直属子节点中必须存在【二级管控部门】。
¢ 如果该一级部门本身被配置为二级管控部门，则视为通过。
 0324更新
1. 对原规则③调整：其直属子节点中必须存在【二级管控部门】→ 其直属子节点中
必须全部是【二级管控部门】
2. 新增规则④：定位到任意部门，其组织路径上必须存在【一级管控部门】和【二级
管控部门】
3. 新增排除表：对于某些特殊部门（如million工作室），不对其进行规则校验

3. 系统交互与触发时机
3.1 弱拦截提醒（配置保存时）
 交互设计：当管理员在前端“预算组织架构树”中修改并点击【保存】时，系统实
时执行一次规则计算。
 反馈机制：若命中上述任一异常规则，系统暂停保存动作，但在页面弹出警告条：
“但检测到预算管控层级配置存在逻辑异常，请检查配置配置是否正确，是否继续
保存？”
3.2 离线巡检与邮件消息通知
 定时任务：每日凌晨系统启动定时任务，对当前全量生效的预算组织树进行扫描。
 消息推送：次日早晨 09:30，系统调用邮件消息网关，向系统管理员发送汇总报
告。

需求三：预算编辑功能按钮重构
1. 业务场景
规范 HRBP 的操作边界，将“调钱”和“调项目”的动作进行隔离。
2. 功能按钮逻辑重塑
2.1 按钮 A：【现有人员/HC 变更】（纯成本调整）
 功能定位：仅用于发起加薪/降薪等调整。
 界面控制：
¢ 在弹出的编辑面板中，不再能修改该员工的【项目标签】（不显示）
¢ 仅开放【工资】字段的调整录入
2.2 按钮 B：【项目标签变更】
 功能定位：用于纠正或变更人员的项目归属池。
 界面控制：弹出的编辑面板仅展示该人员的“当前项目”（可下拉选择新项目标
签）。
 双写逻辑：
系统在点击【确定/保存】动作时，执行两步写入：
a. 写申请单：更新当前正在编辑的预算审批单中，该员工的“申请后”项目属性。
b. 写现状基准：直接调用员工/HC当前预算基准接口，无需等待审批流结束，同
步将该员工在系统内的当前有效“项目标签”进行覆盖修改。
